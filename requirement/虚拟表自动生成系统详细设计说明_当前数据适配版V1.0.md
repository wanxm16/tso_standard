# 面向 Text2SQL 表召回的虚拟表自动生成系统详细设计说明

> **文档版本：** V1.0（当前数据适配版）
> **文档状态：** 建议稿
> **适配对象：** `/Users/Meng/project/tso_standard/data`
> **更新日期：** 2026年4月21日

## 1. 目标重述

当前目标不是“发现已有虚拟表中的标准”，也不是“自动生成可直接执行的大宽表”，而是通过自动化方式，从现有物理表元数据、字段样例、字段使用情况和分类树中，自动生成一批专门服务于 `text2sql` 表召回的虚拟表。

这里的“虚拟表”本质上是语义召回层，而不是物理落表层。它的职责是：

1. 用更接近用户问题表达的方式组织物理表
2. 给 `text2sql` 模型提供更稳定的主题入口
3. 把自然语言问题先路由到少量候选主题表
4. 再通过字段映射回到真实物理表和真实字段

本方案的目标产物是：

1. 自动生成用于表召回的虚拟表主题
2. 自动生成用于语义理解的虚拟字段清单
3. 自动建立虚拟字段到源表字段的映射关系
4. 自动识别主语字段、时间字段、地点字段、关系字段、统计字段
5. 自动为虚拟表补充别名词、问题词和召回标签，支持查询路由
6. 用 benchmark 查询样例持续评估虚拟表是否真的提升表召回效果

一句话概括：系统的目标是“从杂乱物理表中自动沉淀面向 `text2sql` 表召回的语义虚拟表层”。

## 3. 为什么当前更适合做虚拟表自动生成

当前数据有三个很关键的条件：

### 3.1 已有较强字段语义基础

字段注释覆盖率高、字段样例覆盖率高、字段名大量使用拼音全称或稳定缩写，这意味着系统有足够证据把字段自动归一成较稳定的业务概念。

### 3.2 已有较明确的业务主题分类

`二期表分类树.json` 已经给出了较好的主题分组，这对于自动生成虚拟表主题非常关键。虚拟表不应该从“全局所有表”平地起，而应优先从分类树和物理表语义中长出来。

### 3.3 已有真实查询样例可做表召回离线评估

`query_with_table_1.csv` 和 `query_sql.csv` 提供了一个非常重要的离线评估依据：虚拟表生成之后，是否真的能承接真实问题中的主题、字段、筛选条件和输出形式。

这使得虚拟表方案可以从“拍脑袋建模”变成“先自动生成，再用真实问题做效果验证”。

## 6. 总体架构

系统采用“主题发现 + 结构生成 + 映射生成 + 评估回路”的四段式架构。

### 6.1 主题发现层

负责识别哪些业务主题应该形成可被召回的虚拟表。

输入：

1. 分类树
2. 表中文名和表注释
3. 物理表字段分布
4. 字段使用情况

输出：

1. 虚拟表主题候选
2. 主题下候选源表集合

### 6.2 字段归一层

负责把源表字段归一成可供用户问题匹配的业务概念和逻辑类型。

输入：

1. DDL
2. 字段注释
3. 字段样例
4. 字段使用情况

输出：

1. 归一概念
2. 逻辑类型
3. 字段角色
4. 字段质量分

### 6.3 虚拟表生成层

负责生成虚拟表结构、召回标签和字段映射。

输出：

1. 虚拟表定义
2. 虚拟字段清单
3. 字段映射关系
4. 主语字段与时间字段策略
5. 表召回标签与字段别名

### 6.4 评估反馈层

负责利用 benchmark 查询验证虚拟表的表召回价值，并把结果回写到生成策略中。

## 7. 虚拟表生成的核心思想

### 7.1 虚拟表不是“分类 = 表”

不能简单把每个 L2 分类直接变成一张虚拟表。因为：

1. 有些分类更像“主档”
2. 有些分类更像“关系”
3. 有些分类更像“事件流”
4. 有些分类更像“标签或名单”
5. 有些分类内部可能需要拆成多张不同粒度的虚拟表

因此，虚拟表应以“表召回可复用粒度”为中心，而不是纯目录式分组。

### 7.2 虚拟表不是“真正合表”

当前虚拟表的定位是 `text2sql` 表召回层，因此不要求先把多个源表真实合并成一张可执行宽表。

更准确地说，虚拟表是：

1. 一个主题入口
2. 一组召回字段槽位
3. 一组别名与问题表达
4. 一组到物理表和物理字段的映射规则

因此，系统的任务不是“把所有字段并起来”，而是“把多个物理表抽象成一个更容易被用户问题命中的主题表表示”。

### 7.3 虚拟表应优先服务高频查询模式

从当前业务问题类型看，首期最值得优先支持的查询集中在以下类型：

1. 单人核查
2. 关系判断
3. 轨迹事件明细
4. 时间范围统计
5. 主题行为明细

所以虚拟表优先应围绕这些查询模式构建，而不是试图一次性覆盖所有表。

### 7.4 虚拟表的最小单元是“稳定查询粒度”

例如：

1. 人员主档表的粒度是一人一行
2. 婚姻关系表的粒度是一对关系一行
3. 轨迹事件表的粒度是一条轨迹一行
4. 加油行为表的粒度是一次加油事件一行
5. 同行关系表的粒度是一次关系事件或一对关系汇总一行

## 8. 自动生成流程

当前版本建议按如下流程生成虚拟表：

1. 解析分类树，得到初始主题集合
2. 解析物理表语义，补充主题候选
3. 以“分类主题 + 表语义 + 字段分布”融合生成候选主题
4. 对所有字段进行语义归一与逻辑类型推断
5. 自动把字段归并到候选虚拟字段槽位
6. 只对冲突项和低置信项生成 review 队列
7. 在每个主题内识别候选主体表、明细表、补充表
8. 判断虚拟表类型与粒度
9. 生成核心字段集、展示字段集、筛选字段集和别名词包
10. 为每个虚拟字段选择最佳源字段或生成映射规则
11. 生成虚拟表定义草案和召回描述
12. 用 benchmark 查询对虚拟表进行离线评估
13. 根据评估结果合并、拆分、重命名或淘汰虚拟表

## 9. 主题发现策略

主题发现采用三路信号融合：

### 9.1 分类树信号

分类树提供“应该有哪几类主题”的初始边界。

### 9.2 物理表语义信号

表中文名和表注释可以帮助识别主题类型，例如：

1. `最近六个月加油信息` 明显偏事件表
2. `最后一次出入境记录` 明显偏单人最新状态表
3. `学生信息表` 偏实体主档

### 9.3 字段结构与使用信号

字段分布、关键标识、时间字段、地点字段以及字段使用情况，可以帮助判断一个主题是否应沉淀为单独虚拟表，以及应采用什么粒度。

### 9.4 融合结果

最终主题应不是简单字符串，而应包含：

1. `topic_name`
2. `topic_source`
3. `candidate_tables`
4. `topic_confidence`
5. `suggested_table_type`

## 10. 字段归一与字段角色识别

字段归一是虚拟表生成的基础。当前版本建议对每个字段识别三类信息：

1. 它表达的是什么概念
2. 它在虚拟表里扮演什么角色
3. 它是否足够重要到进入虚拟表

这里的“归一”不是把字段机械压成一个统一列名，而是把字段归入一个用户可理解的语义字段槽位。

例如：

1. `sfzh`、`gmsfhm`、`zheng_jian_hao` 可以归入 `certificate_no`
2. `xm`、`xing_ming`、`crj_xing_ming` 可以归入 `person_name`
3. `chu_ru_jing_shi_jian`、`sj`、`jin_ru_shi_jian` 可以按主题分别归入 `event_time`

同一个语义槽位下面允许存在多个来源字段映射，而不要求只有一个唯一物理来源。

### 10.1 归一化总体策略

当前版本建议采用“系统自动归一化为主，人工审核为辅”的工作模式。

目标不是让人工从零开始整理字段，而是让系统先自动完成以下工作：

1. 识别字段语义概念
2. 识别逻辑类型
3. 识别字段角色
4. 给出候选虚拟字段槽位
5. 给出来源映射与置信度
6. 标记需要人工介入的冲突项

人工主要负责审核以下少量高风险情况：

1. 同名不同义
2. 同义不同角色
3. 多个高置信候选相互冲突
4. 低置信度字段无法自动归槽
5. 影响高频查询召回效果的核心字段

也就是说，人工 review 的对象应是“系统挑出来的少量问题点”，而不是“全量字段”。

### 10.2 自动归一化流水线

建议把自动归一化拆成五步：

#### 第一步：字段预处理

对每个源字段提取基础特征：

1. 字段名 token
2. 注释文本
3. 样例模式
4. 使用热度
5. 所在表语义
6. 所属分类

#### 第二步：字段语义判定

对每个字段输出：

1. `canonical_concept`
2. `logical_type`
3. `field_role`
4. `aliases`
5. `confidence`

#### 第三步：候选槽位归并

把多个字段自动归并到候选虚拟字段槽位中，例如：

1. `sfzh`、`gmsfhm`、`zheng_jian_hao` 归到 `certificate_no`
2. `xm`、`xing_ming`、`crj_xing_ming` 归到 `person_name`
3. `chu_ru_jing_shi_jian`、`jin_ru_shi_jian` 归到不同主题下的 `event_time`

#### 第四步：冲突检测

系统自动检测以下问题：

1. 一个字段可归入多个槽位
2. 一个槽位内逻辑类型不一致
3. 一个槽位内字段角色冲突
4. 同一主题下出现多个极相近槽位

#### 第五步：生成 review 队列

只把冲突项和低置信项送审，其余字段直接进入草案结果。

### 10.3 需要识别的字段角色

建议至少识别以下角色：

1. 主语字段
2. 关系主语字段
3. 主体标识
4. 展示字段
5. 时间字段
6. 地点字段
7. 统计字段
8. 过滤字段
9. 来源字段
10. 描述字段

### 10.4 字段重要性评分

字段重要性建议综合以下因素：

1. 字段使用次数
2. 字段查询次数
3. `role_select` 权重
4. `role_where` 权重
5. `role_join` 权重
6. 字段语义清晰度
7. 字段样例质量

重要性高的字段优先进入虚拟表主字段集。

同时，字段进入虚拟表时还应额外生成：

1. 中文别名集合
2. 常见问题表达
3. 同义问法关键词

因为对表召回而言，字段能否被用户问题命中，和字段本身的别名覆盖度同样重要。

### 10.5 归一化置信度分层与审核策略

建议把自动归一化结果按置信度分三层处理：

1. 高置信
2. 中置信
3. 低置信

建议规则如下：

1. 高置信：直接进入虚拟字段草案，仅抽样 review
2. 中置信：进入待审队列，由人工快速确认或改槽
3. 低置信：不自动入正式草案，必须人工确认

一个可落地的经验阈值可以是：

1. `confidence >= 0.85`：高置信
2. `0.60 <= confidence < 0.85`：中置信
3. `confidence < 0.60`：低置信

这样可以把 review 的工作量控制在最值得看的那部分。

### 10.6 review 页面建议关注的信息

为了让人工 review 高效，建议 review 页面直接展示：

1. 候选虚拟字段槽位名
2. 源字段名
3. 源字段注释
4. 样例值
5. 所在表名与分类
6. 自动归一结果
7. 置信度
8. 冲突原因
9. 备选槽位

review 操作建议尽量简单：

1. 确认归一结果
2. 改归一槽位
3. 拆分槽位
4. 合并槽位
5. 标记例外
6. 反馈新别名

### 10.7 当前场景中的高价值逻辑类型

建议重点支持：

1. `person_name`

3. `passport_no`
4. `phone_no`

### 10.8 自动归一化的具体实现方法

当前版本建议采用“规则优先 + 统计打分 + LLM 兜底 + 人工只审冲突项”的自动归一化方案。

这套方案的目标不是让模型自由发挥，而是让系统在大多数字段上给出可解释、可复现、可回溯的归一化结果。

#### 10.8.1 归一化范围控制

自动归一化不建议在全库范围一次性执行，而应限制在“同一主题、同一粒度”的候选源表集合中进行。

例如：

1. “出入境记录”主题内的多个事件表可以一起归一
2. “人员主档”主题内的多张主档表可以一起归一
3. “婚姻关系”与“出入境记录”不应直接放在同一轮归一

这样做的好处是：

1. 候选槽位更稳定
2. 同义字段更容易收敛
3. 冲突项显著减少
4. review 成本更低

#### 10.8.2 每个字段需要抽取的特征

建议对每个字段抽取六类特征：

1. 名称特征
2. 注释特征
3. 样例特征
4. 使用特征
5. 上下文特征
6. 表语义特征

名称特征包括：

1. 原始字段名
2. 下划线切词结果
3. 拼音切词结果
4. 缩写展开结果
5. 后缀模式，如 `_id`、`_no`、`_sj`

注释特征包括：

1. 原始注释
2. 注释分词
3. 注释归一结果
4. 注释命中的逻辑类型关键词

样例特征包括：

1. 是否符合身份证号模式
2. 是否符合手机号模式
3. 是否符合日期或日期时间模式
4. 是否像行政区划码、车牌号、护照号
5. 值长度分布和字符分布

使用特征包括：

1. `usage_count`
2. `query_count`
3. `role_select`
4. `role_where`
5. `role_join`

上下文特征包括：

1. 同表相邻字段
2. 同表是否共现主语字段
3. 同表是否共现时间字段、地点字段、状态字段

表语义特征包括：

1. 表名
2. 表中文名
3. 表注释
4. 所属分类
5. 当前候选主题

#### 10.8.3 预处理与标准化

在正式归一化前，建议先做字段级预处理。

预处理步骤如下：

1. 清洗技术噪声后缀，如 `new`、`result`、`tmp`、`df`
2. 清洗明显技术字段，如 `rn`、`dt`、`etl_time`
3. 对拼音字段做切词和缩写展开
4. 对注释进行同义归一，例如“公民身份号码”和“身份证号”
5. 对样例值提取模式标签

例如：

1. `sfzh` -> 候选概念：“身份证号”
2. `gmsfhm` -> 候选概念：“公民身份号码”
3. `zheng_jian_hao` -> 候选概念：“证件号”

这三者在主题上下文支持下，可以进一步归到统一槽位 `certificate_no`。

#### 10.8.4 语义字段槽位生成

自动归一化的目标不是直接把字段两两合并，而是把字段归入一组预定义或半自动生成的语义字段槽位。

建议每个主题至少维护以下几类槽位：

1. 主语槽位
2. 标识槽位
3. 时间槽位
4. 地点槽位
5. 行为或关系槽位
6. 状态槽位
7. 来源槽位
8. 描述槽位
9. 统计槽位

例如“出入境记录”主题下，可以自动长出如下槽位：

1. `person_name`
2. `certificate_no`
3. `entry_exit_time`
4. `port_name`
5. `direction`
6. `destination_country`
7. `transport_mode`
8. `source_system`

#### 10.8.5 字段到槽位的打分公式

建议对每个“字段 -> 槽位”的候选关系进行可解释打分。

一个可落地的基线公式如下：

```text
slot_score =
  0.30 * lexical_score
+ 0.25 * comment_semantic_score
+ 0.20 * sample_pattern_score
+ 0.15 * context_role_score
+ 0.10 * usage_score
```

其中：

1. `lexical_score`
   字段名、拼音展开、缩写展开与槽位别名的接近程度
2. `comment_semantic_score`
   字段注释与槽位描述、槽位别名之间的语义接近程度
3. `sample_pattern_score`
   样例值是否符合该槽位对应逻辑类型
4. `context_role_score`
   字段在当前主题中的角色是否符合槽位角色
5. `usage_score`
   字段在真实使用中是否体现出高价值，例如常用于 `where`、`join`

#### 10.8.6 每个子分数的建议计算方法

`lexical_score` 建议综合：

1. token overlap
2. 编辑距离相似度
3. 缩写词典命中
4. 槽位别名命中

`comment_semantic_score` 建议综合：

1. 注释关键词精确命中
2. 注释同义词命中
3. 向量相似度或轻量文本匹配

`sample_pattern_score` 建议综合：

1. 正则匹配
2. 样例值长度分布
3. 数字/中文/字母构成
4. 枚举值形态

`context_role_score` 建议综合：

1. 同表是否已有主语字段
2. 当前字段是否与时间、地点、行为词共现
3. 当前主题是否天然偏主档/关系/事件

`usage_score` 建议综合：

1. `role_where` 权重
2. `role_join` 权重
3. `role_select` 权重
4. `query_count`

#### 10.8.7 自动归槽决策规则

建议按如下规则自动做第一轮归槽：

1. 若第一名槽位分数 `>= 0.85` 且领先第二名 `>= 0.10`，直接自动归入
2. 若第一名分数在 `0.60 ~ 0.85`，标记为中置信，进入 review 队列
3. 若第一名分数 `< 0.60`，标记为低置信，不自动归槽
4. 若前两名槽位分差过小，标记为冲突项

这样可以把大部分清晰字段自动处理掉，只把真正模糊的字段留给人工审核。

#### 10.8.8 典型冲突检测规则

系统应自动检测以下冲突：

1. 同名不同义
   例如 `type` 在不同主题中可能表示完全不同含义
2. 同义不同角色
   例如“本人身份证号”和“关系人身份证号”
3. 同槽位内逻辑类型冲突
   例如有的值像身份证号，有的值像护照号
4. 同主题下存在多个高度相似槽位
   例如 `event_time` 和 `occur_time`

这些冲突不应自动沉默处理，而应进入 review 队列。

#### 10.8.9 LLM 的使用边界

LLM 不建议直接决定最终归一化结果，而应只在以下场景做兜底：

1. 字段名高度缩写且注释缺失
2. 规则和样例判断冲突
3. 注释很长但语义模糊
4. 候选槽位之间分差很小

LLM 输出建议只作为附加证据，进入 `evidence` 字段，不直接覆盖规则分数。

#### 10.8.10 推荐产出结构

建议系统对每个字段输出结构化归一化结果，便于 review 和后续回溯。

```json
{
  "topic": "出入境记录",
  "source_table": "ywk_sc_sj_crjjl_zjbl_zh_sfzh_last_2001_lhcz_ai_model_test",
  "source_field": "zheng_jian_hao",
  "canonical_concept": "证件号",
  "logical_type": "passport_no_or_id_card_no",
  "candidate_slots": [
    {"slot": "certificate_no", "score": 0.91},
    {"slot": "person_id", "score": 0.22}
  ],
  "selected_slot": "certificate_no",
  "field_role": "subject_identifier",
  "confidence": 0.91,
  "review_status": "auto_accepted",
  "evidence": {
    "comment": "证件号",
    "sample_pattern": "X02008986",
    "aliases_hit": ["证件号", "护照号", "身份证号"],
    "context_fields": ["出入境时间", "出入境口岸", "前往国家"]
  }
}
```

#### 10.8.11 反馈回写机制

人工 review 不是重新做归一化，而是修正系统结果。所有修正都应回写到以下资产中：

1. 缩写词典
2. 注释同义词典
3. 槽位别名词包
4. 正负样本集
5. 冲突规则

这样才能让下一轮自动归一化越来越稳，而不是每轮都重复人工劳动。

#### 10.8.12 伪代码示意

下面给一版接近实现的伪代码：

```python
for topic in topics:
    source_fields = load_fields(topic)
    slot_defs = build_candidate_slots(topic)

    for field in source_fields:
        features = extract_features(field)
        concept = infer_canonical_concept(features)
        logical_type = infer_logical_type(features)
        role = infer_field_role(features, topic)

        slot_scores = []
        for slot in slot_defs:
            score = (
                0.30 * lexical_score(field, slot)
                + 0.25 * comment_semantic_score(field, slot)
                + 0.20 * sample_pattern_score(field, slot)
                + 0.15 * context_role_score(field, slot, topic)
                + 0.10 * usage_score(field)
            )
            slot_scores.append((slot.name, score))

        slot_scores.sort(key=lambda x: x[1], reverse=True)
        top1, top2 = slot_scores[0], slot_scores[1] if len(slot_scores) > 1 else (None, 0.0)

        if top1[1] >= 0.85 and top1[1] - top2[1] >= 0.10:
            review_status = "auto_accepted"
            selected_slot = top1[0]
        elif top1[1] >= 0.60:
            review_status = "needs_review"
            selected_slot = top1[0]
        else:
            review_status = "low_confidence"
            selected_slot = None

        save_normalization_result(
            topic=topic,
            field=field,
            concept=concept,
            logical_type=logical_type,
            role=role,
            slot_scores=slot_scores,
            selected_slot=selected_slot,
            review_status=review_status,
        )
```

这版流程的关键优点是：

1. 大部分字段可自动归一
2. 所有结果都有证据和分数
3. review 范围被控制在少量疑难项
4. 非常适合你们当前“自动化为主、人工辅助”的目标

## 11. 虚拟表类型判定

当前版本建议优先支持五类虚拟表：

### 11.1 主档类虚拟表

适用于一人一行、一车一行、一设备一行。

示例：

1. 人员主档
2. 车辆主档
3. 地址主档

### 11.2 关系类虚拟表

适用于两主体之间的关系。

示例：

1. 婚姻关系
2. 亲属关系
3. 同行关系
4. 同监区关系

### 11.3 事件类虚拟表

适用于一次行为、一条轨迹、一条记录。

示例：

1. 出入境记录
2. 加油行为
3. 住宿记录
4. 轨迹事件

### 11.4 标签类虚拟表

适用于名单、标签、重点人员等。

### 11.5 聚合类虚拟表

适用于最近一次、近一年汇总、多源汇总等面向查询的结果层。

说明：聚合类虚拟表不建议一开始就大规模自动生成，应先按明确业务主题按需生成，再交由 benchmark 做效果验证。

## 12. 源表选择策略

不是所有进入主题的表都应该成为虚拟表的数据源。

建议对源表打分：

1. 与主题名称相似度
2. 字段完整性
3. 关键标识字段是否齐全
4. 时间字段是否齐全
5. 地点字段是否齐全
6. 数据使用热度
7. 是否存在大量无注释或技术字段

源表按角色分为：

1. 主源表
2. 补充表
3. 关系补全表
4. 统计补全表

## 13. 虚拟字段生成策略

虚拟字段生成要遵循“少而稳”的原则。

每张虚拟表的字段应优先分三层：

### 13.1 必备字段

例如：

1. 主语字段
2. 主体标识
3. 主题核心属性
4. 时间字段
5. 来源字段

### 13.2 高频展示字段

通过字段使用情况、`role_select`、`role_where` 和表注释推断。

### 13.3 可选扩展字段

用于后续增强，不作为首版主字段。

### 13.4 字段映射类型

建议支持以下映射类型：

1. 直接映射
2. 同义字段择优
3. 多字段合并
4. 字段拆分
5. 规则计算
6. 最新值提取
7. 聚合统计

说明：对 `text2sql` 表召回来说，字段映射不需要在这一层完全落成执行 SQL，只需要明确“这个虚拟字段通常应下钻到哪些真实字段”。

## 14. 主键与粒度推断

虚拟表自动生成中，最关键的是自动判断表粒度。

### 14.1 主档类表

如果表中存在身份证号、人员编号、车牌号、设备号等稳定标识，并且时间字段不是主分区语义，则倾向主档类。

### 14.2 事件类表

如果表中同时存在“主体标识 + 时间 + 地点/行为”，则倾向事件类。

### 14.3 关系类表

如果表中存在“两端主体标识 + 关系类型/关系事实”，则倾向关系类。

### 14.4 聚合类表

如果表名或注释中有“最近一次”“最后一条”“汇总”“整合”等词，且同主体只保留少量行，则倾向聚合类。

## 15. benchmark 驱动的评估机制

当前版本必须引入 benchmark 作为闭环，否则虚拟表生成容易停留在“看起来合理”。

### 15.1 评估目标

判断生成的虚拟表是否能承接 `text2sql` 表召回的核心需求。

### 15.2 建议评估指标

1. `topic_hit_rate`
2. `table_recall_at_k`
3. `virtual_field_hit_rate`
4. `query_support_rate`
5. `sql_rewrite_success_rate`

### 15.3 查询支持判定

若某条 benchmark 查询所需的：

1. 问题意图能命中某张虚拟表
2. 主过滤字段能在虚拟表中找到
3. 主输出字段能在虚拟表中找到
4. 时间条件可以表达

则认为该虚拟表对这条查询“基本可支持”。

对于表召回而言，最重要的不是“字段是否完全一致”，而是：

1. 正确主题能否进入 TopK
2. 问题中的关键实体、时间、地点是否能在虚拟表字段槽位中找到对应
3. 后续 SQL 生成是否知道应该回落到哪些物理表

## 16. 当前版本的落地策略

### 16.1 第一阶段：自动生成候选虚拟表草案

输出：

1. 主题
2. 表类型
3. 候选字段
4. 源表映射
5. 评分

### 16.2 第二阶段：人工审核少量高价值主题

优先审核：

1. 人员主档
2. 轨迹事件
3. 加油行为
4. 出入境记录
5. 婚姻关系
6. 同监区关系

### 16.3 第三阶段：用 benchmark 验证并迭代

每轮生成后重新跑 benchmark，保留提升明显的虚拟表，淘汰冗余主题。

## 17. 当前版本不建议做的事情

1. 一开始就自动生成全量 SQL 视图
2. 一开始就尝试复杂多跳 join 编排
3. 一开始就生成过宽的大宽表
4. 一开始就完全依赖 LLM 自由命名和自由建模
5. 不经过 benchmark 验证就大量产出虚拟表

## 18. 推荐的最小可行产物

如果按 MVP 落地，建议首期只交付三样东西：

1. 一份自动生成的虚拟表目录
2. 每张虚拟表的字段、别名与源表映射 JSON
3. 一份 benchmark 表召回效果报告

这样最容易判断方案是否真的有效，也最方便后续继续自动化。

## 19. 最终建议

对于当前数据，最合适的路线不是“先做标准库，再慢慢想虚拟表”，而是：

1. 以分类树和物理表语义为主题入口
2. 以字段归一、字段角色识别和字段别名扩展为底层能力
3. 以自动生成“表召回语义层”虚拟表为直接目标
4. 以 benchmark 离线评估为质量闭环

因此，当前最适合你们数据的方案，是“面向 `text2sql` 表召回的虚拟表自动生成方案”，而不是上一版偏“标准发现”的方案，也不是面向执行层的大宽表方案。
